    Введение

    Добро пожаловать в REST API Единой Биллинговой Системы (ЕБС).

    Здесь вы найдете всю информацию, чтобы начать работу с нашим API. API организован по стандарту REST. Наш API имеет предсказуемые, ориентированные на ресурсы URL-адреса и использует HTTP-коды ответов для указания ошибок API. Мы используем встроенные функции HTTP, такие как HTTP-аутентификация и HTTP-глаголы, которые понятны для готовых HTTP-клиентов. JSON возвращается всеми ответами API, включая ошибки.

    Среды

    Этот API доступен в 2 средах:

    • Production API. URL: https://bapi.birdarcha.uz
    • Sandbox API для тестирования. URL: http://91.212.89.229:8981

    Инструменты

    Swagger UI - позволяет визуализировать ресурсы API и взаимодействовать с ними без какой-либо логики реализации.

    • Swagger для Production
    • Swagger для Sandbox

    Аутентификация

    В целях безопасности мы требуем, чтобы приложения запрашивали и получали разрешение на доступ к любой конкретной учетной записи (даже если она ваша). Это делается с использованием стандартного протокола OAuth 2.0.

    OAuth 2.0 основан на использовании базовых веб-технологий: HTTP-запросах, редиректах и т. п. Поэтому использование OAuth возможно на любой платформе с доступом к интернету и браузеру: на сайтах, в мобильных и desktop-приложениях, плагинах для браузеров…

    Общая схема работы приложения, использующего OAuth, такова:

    • получение авторизации
    • обращение к защищенным ресурсам

    Результатом авторизации является access token — некий ключ (обычно просто набор символов), предъявление которого является пропуском к защищенным ресурсам. Обращение к ним в самом простом случае происходит по HTTPS с указанием в заголовках или в качестве одного из параметров полученного access token.

    Регистрация в ЕБС

    Для получения токена доступа вам понадобится login и password также client_id и client_secret.

    Обратите к системному интегратору ЕБС о предоставлении им доступа. Системный интегратор в свою очередь добавляет ваша приложения в биллинговой системе. После регистрации приложения, OAuth-сервер ЕБС сгенерирует login и password также client_id и client_secret параметры, которые нужно будет указать при получени токена.

    Получение токена

    Получив login и password также client_id и client_secret, вы можете обменять их на токен доступа.

    Пример запроса получение токена:

    curl -X GET http://91.212.89.229:8981/oauth/token?grant_type=password&username=test&password=123
-H 'Accept: application/json'
-H 'Authorization: Basic <base64-encoded-string>'

    curl -X GET https://bapi.birdarcha.uz/oauth/token?grant_type=password&username=test&password=123
-H 'Accept: application/json'
-H 'Authorization: Basic <base64-encoded-string>'

    Параметры заголовки

    Обозначение Наименование Примечание
    Authorization base64(<client_id>:<client_secret>), закодированная методом base64

    Параметры запроса

    Обозначение Наименование Примечание
    grant_type тип ответа Обязательный параметр, параметру grant_type всегда присваивается значение login и password
    username логин пользователя Обязательный параметр
    password пароль пользователя Обязательный параметр

    Пример ответа в случае успеха:

    {
  "access_token":"13e32e6a-f54d-4962-98d6-dd219c73b0a7",
  "token_type":"bearer",
  "refresh_token":"b6a2a361-4e9c-4d27-b3e4-a42e982417fa",
  "expires_in":1199,
  "scope":"read write",
  "status":"ok"
}

    Параметры ответа:

    Обозначение Наименование Примечание
    access_token токен токен, с которым приложение может работать от имени пользователя.
    token_type тип ключа определяющий как его надо использовать
    refresh_token токен для обновление
    expires_in время жизни токена доступа Значение указывается в секундах.
    scope областью видимости Доступ приложения к пользовательскому аккаунту ограничен "областью видимости" (scope) предоставленных прав авторизации (например, доступ на чтение или запись)
    status статус ответа ok если успешно

    Пример ответа в случае ошибки:

    {
  "error":"invalid_request"
}

    HTTP-код ответа может быть 400, 401. В теле ответа помимо error могут быть дополнительные параметры, например:

    • invalid_request - неверный формат запроса
    • invalid_grant - неверный или просроченный код подтверждения
    • unsupported_grant_type - неверное значение параметра grant_type -->
    Все запросы к методам API должны содержать в заголовках `Authorization Bearer `

    Восстановление авторизации

    Обычно, access token имеет ограниченный срок годности. Чтобы не заставлять пользователя проходить авторизацию после истечения срока действия access token, во всех перечисленных выше вариантах, в дополнение к access token может возвращаться еще refresh token. По нему можно получить access token с помощью HTTP-запроса, аналогично авторизации по логину и паролю.

    Пример обновление токена:

    curl -X GET http://91.212.89.229:8981/oauth/token?grant_type=refresh_token&refresh_token=b6a2a361-4e9c-4d27-b3e4-a42e982417fa
-H 'Accept: application/json'
-H 'Authorization: Basic <base64-encoded-string>'

    curl -X GET https://bapi.birdarcha.uz/oauth/token?grant_type=refresh_token&refresh_token=b6a2a361-4e9c-4d27-b3e4-a42e982417fa
-H 'Accept: application/json'
-H 'Authorization: Basic <base64-encoded-string>'
    Обозначение Наименование Примечание
    Authorization base64(<client_id>:<client_secret>), закодированная методом base64

    Параметры запроса

    Обозначение Наименование Примечание
    grant_type тип ответа Обязательный параметр. Параметр grant_type=refresh_token, а параметр refresh_token должен содержать значение токена перевыпуска
    refresh_token токен для обновления токена доступа Обязательный параметр.

    Формат ответа в случае успеха:

    {
  "access_token":"13e32e6a-f54d-4962-98d6-dd219c73b0a7",
  "token_type":"bearer",
  "refresh_token":"b6a2a361-4e9c-4d27-b3e4-a42e982417fa",
  "expires_in":1199,
  "status":"ok"
}
    Все запросы к методам API должны содержать в заголовках `Authorization Bearer `

    Отмены инвойса

    Ранее созданный инвойс можно отменить, если он еще не был оплачен. Для этого в методе `DELETE /api/invoice` нужно указать параметр id из ранее сформированного инвойса.

    Пример отмены инвойса:

    curl -X GET http://91.212.89.229:8981/api/invoice?id=1841974 \
-H 'Accept: application/json' \
-H 'Authorization: Bearer 44e07a7d-3950-4e06-b6cf-2eee951b8a8d'

    curl -X GET https://bapi.birdarcha.uz/api/invoice?id=1841974 \
-H 'Accept: application/json' \
-H 'Authorization: Bearer 44e07a7d-3950-4e06-b6cf-2eee951b8a8d' \

    Request

    DELETE https://bapi.birdarcha.uz/api/invoice?id=

    В случае успеха будет возвращен true.

    Получить инвойс

    Ранее созданный инвойс можно получить по его серийному номеру (serial).

    Пример запрос получить инвойса:

    curl -X GET http://91.212.89.229:8981/api/invoice/{serial} \

    Errors

    The Kittn API uses the following error codes:

    Error Code Meaning
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your API key is wrong.
    403 Forbidden -- The kitten requested is hidden for administrators only.
    404 Not Found -- The specified kitten could not be found.
    405 Method Not Allowed -- You tried to access a kitten with an invalid method.
    406 Not Acceptable -- Insufficient value
    409 Conflict
    417 Expectation Failed
    429 Too Many Requests -- You're requesting too many kittens! Slow down!
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
